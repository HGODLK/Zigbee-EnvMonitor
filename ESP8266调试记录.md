# ESP8266 WiFi 模块调试记录

## 硬件连接

| ESP8266 | CC2530 板1 (协调器) |
|---------|-------------------|
| RX | P0_5 (TX1, UART1 Alt 1) |
| TX | P0_4 (RX1, UART1 Alt 1) |
| RST | 板子 RST |
| EN | 板子 EN |
| IO0 | NC (悬空) |
| IO2 | PI (板子标注) |

IGT 复位控制：P0_6（低电平复位）

## 已确认正常

- OLED 显示、S1 按键切换、Zigbee 组网、DHT11 温湿度、光敏传感器 ✅
- ESP8266 **蓝色 LED 闪烁**（启动成功）✅
- CC2530 **能收到 ESP8266 启动信息**（74880 波特时发送的 boot message）→ 说明 UART1 RX 通路正常 ✅

## 问题

**ESP8266 不回复 AT 命令**。CC2530 通过 UART1 发送 `AT\r\n` 或 `AT+CWMODE=1\r\n`，ESP8266 没有任何响应。

串口日志（9600, UART0 debug）：
```
>> AT
[ESP] S=1
CWMODE timeout, buf:     ← 2秒超时，缓冲区为空
[ESP] S=8 → S=0 → 循环
```

## 已尝试的方案

| 尝试 | 结果 |
|------|------|
| UART1 Alt 1 (P0_4/P0_5) 115200 | 收到启动信息，AT 无回复 |
| UART1 Alt 2 (P1_4/P1_5) 115200 | 完全无通信 |
| UART1 9600 波特 | 乱码启动信息，AT 无回复 |
| UART1 115200 (匹配示例) | 启动信息通，AT 无回复 |
| U1CSR = 0xC0 (开 RX) | 同上 |
| U1CSR \|= 0x80; \|= 0x40 (分步) | 同上 |
| IEN0 \|= 0x88 (开中断) | 同上 |
| P2DIR &= ~0xC0 (优先级) | 板子无输出(可能干扰UART0) |
| 复位脉冲 50ms → 1s | 同上 |
| 先发 AT 再发 AT+CWMODE | 同上 |

示例路径：`ZIGBEE开发套件WiFi模块全部资料V3.0\6.代码及其教程\2.进阶篇-TI BasicRF 简单无线点对点传输协议\3.WiFi模块ESP8266进行网络数据传输\3.ZIGBEE驱动WiFi模块ESP8266串口显示WiFi模块SSID和WiFi模块的IP地址`

示例 `initUART1()`：
```c
void initUART1(void) { 
    PERCFG = 0x00;       // UART1 Alt 1: P0_4/P0_5
    P0SEL |= 0x30;       // P0_4,P0_5 as peripheral
    U1CSR |= 0x80;       // UART mode
    U1GCR |= 11;         // BAUD_E = 11
    U1BAUD |= 216;       // BAUD_M = 216 → 115200 @ 32MHz
    UTX1IF = 0;          // Clear TX flag
    U1CSR |= 0X40;       // Enable RX
    IEN0 |= 0x88;        // Enable interrupts
}
```

我们的 `u1_init()` 已逐行匹配，但仍不工作。

## 核心怀疑

1. **CC2530 TX (P0_5) 没发出去** — 虽然 UART1 配置正确，但 `U1CSR & 0x02` (TX_BYTE) 可能一直不置位，`u1_tx()` 超时退出，ESP8266 根本没收到的 AT 命令
2. **Z-Stack HAL 干扰了 UART1** — Z-Stack 的 `hal_board_cfg.h` 中有 UART1 SPI 模式配置，可能与我们的 UART 模式冲突
3. **ESP8266 RX 引脚未正确连接** — 虽然在"TX1/RX1"丝印上，但实际 PCB 走线可能不同

## 下一步建议

- 用示波器/逻辑分析仪抓 P0_5 波形，确认 AT 命令是否发出
- 用 USB-TTL 直接连 ESP8266 确认模块 AT 固件正常
- 检查 Z-Stack HAL 是否重新配置了 UART1 (PERCFG/U1CSR)
